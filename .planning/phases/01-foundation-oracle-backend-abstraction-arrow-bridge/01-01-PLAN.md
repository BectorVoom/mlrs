<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| host slice → CubeCL device buffer | Host `&[f32]`/`&[f64]` crosses into a device buffer via `client.create` / `Bytes`; length/alignment must be correct or `unsafe { ArrayArg::from_raw_parts }` reads OOB. |
| Cargo feature selection → linked runtime | Wrong/multiple backend features could link conflicting runtimes; `mlrs-kernels` must stay feature-free. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | `spike_test.rs` `unsafe { ArrayArg::from_raw_parts }` with wrong length | mitigate | Always derive length from the validated slice `.len()`; kernel bounds-check `if tid < x.len()`. |
| T-01-02 | Denial of Service | `mlrs-kernels` accidentally pulling a backend runtime feature (breaks portability/build) | mitigate | `cubecl` with `default-features = false`, no runtime feature; CI `cargo tree -p mlrs-kernels` gate (Criterion 1). |
| T-01-03 | Tampering | Multiple backend features active simultaneously selecting conflicting `ActiveRuntime` | mitigate | Exactly-one-feature contract in `runtime.rs` via `cfg` re-exports; document in SPIKE-FINDINGS.md. |
| T-01-SC | Tampering | cargo installs (cubecl/arrow/bytemuck/npz reader/logger) | mitigate | Package Legitimacy Audit in RESEARCH.md (all first-party, `[ASSUMED]` only because slopcheck unavailable) — see Plan 02 blocking checkpoint for the npz reader before its first real use. |
</threat_model>
