<objective>
Build the one genuinely-new device primitive of Phase 4: a Cholesky
factorization + forward/back triangular solve for a small SPD system
`A·x = b` (`A = XᵀX + αI`, `n = n_features ≤ MAX_DIM = 64`), required by Ridge
(D-02). This is the single highest implementation risk in the phase — it has no
Phase-2/3 analogue — but it follows the proven `jacobi_eig.rs` single-cube,
all-shared-memory, in-kernel blueprint exactly.

It is validated STANDALONE (f32+f64, cpu+rocm) against a scipy reference plus the
algebraic invariants `‖L·Lᵀ − A‖` and `‖A·x − b‖` BEFORE Ridge (04-05) consumes
it, mirroring Phase-2/3 primitive-first discipline.

Purpose: Ridge's normal-equations solver needs an SPD solve that does not exist
in the primitive set. Build exactly the one SPD solve Ridge needs — not a
general LU/pivoted layer.
Output: `mlrs-kernels::cholesky` kernel; `prims::cholesky` host wrapper;
activated `cholesky_test.rs`.
</objective>

  <action>
    Create `crates/mlrs-kernels/src/cholesky.rs` with a feature-free `#[cube(launch)] pub fn cholesky_solve<F: Float + CubeElement>(a_in: &Array<F>, b_in: &Array<F>, x_out: &mut Array<F>, l_out: &mut Array<F>, info_out: &mut Array<F>, n: u32, rhs: u32)`. The kernel writes the lower Cholesky factor `L` (row-major `n×n`, strictly-upper entries left at 0) into the dedicated `l_out` output buffer in addition to writing the solution into `x_out` — exposing `L` explicitly is REQUIRED so the host can check the `‖L·Lᵀ − A‖` invariant without re-deriving the factor (resolves the prior "Claude's Discretion" ambiguity). Reuse the exported `MAX_DIM` (64) from `mlrs-kernels` (already consumed by `eig.rs`); the host rejects `n > MAX_DIM` before launch. Stage A into `SharedMemory::<F>::new((MAX_DIM * MAX_DIM) as usize)` sized to the comptime cap (jacobi_eig pattern).

    Three in-kernel phases with `sync_cube()` between each (start with the simplest correct unit-0-does-all schedule per RESEARCH Open Q2, since n ≤ 64 makes serialization cheap):
    1. Cholesky-Banachiewicz: for `i in 0..n`, `j in 0..=i`: `L[i][j] = (A[i][j] − Σ_{k<j} L[i][k]·L[j][k]) / L[j][j]` for j<i; `L[i][i] = sqrt(A[i][i] − Σ_{k<i} L[i][k]²)`. Write each computed `L[i][j]` into BOTH the shared tile (for the solve phases) AND `l_out[i*n + j]` (for the host invariant check). GUARD the diagonal sqrt argument: if it is `≤ NEAR_ZERO_FLOOR` set `info_out[0]` to a negative flag and do NOT emit NaN (RESEARCH Pitfall 4). Use `if`-wrap, NOT `continue` (unsupported in `#[cube]`).
    2. Forward solve `L·z = b` (per rhs column): `z[i] = (b[i] − Σ_{k<i} L[i][k]·z[k]) / L[i][i]`.
    3. Back solve `Lᵀ·x = z`: for `i` descending, `x[i] = (z[i] − Σ_{k>i} L[k][i]·x[k]) / L[i][i]`.

    NO hardcoded plane width / 32 — use the shared-memory layout, not a plane path (carried no-hardcoded-plane-width rule). Use generic constants via `F::from_int(..)` / `F::new(..)` and `Float` methods `.sqrt()`/`.abs()`. Add `pub mod cholesky;` to `crates/mlrs-kernels/src/lib.rs`. Copy the `jacobi_eig.rs` module-doc style and the trailing `// tests live in crates/mlrs-backend/tests/cholesky_test.rs` note (NO in-source mod tests, AGENTS.md §2).
  </action>

  </verify>
