<objective>
Close the two verification gaps in Phase 10's MBSGDClassifier `optimal`-schedule
coverage (10-VERIFICATION.md, truth #2 FAILED; CR-01 BLOCKER + WR-01 WARNING from
10-REVIEW.md).

GAP 2 (WR-01) is fixed FIRST because it changes the solver iterate that GAP 1's
new oracle test asserts against. The L2 `wscale` penalty shrink in `sgd_solve` is
currently applied BEFORE the gradient step; RESEARCH.md §"Per-sample update
sequence" (lines 341-348) and §Penalty (line 231: "applied AFTER the gradient
step") AND sklearn's `_plain_sgd` all specify it AFTER. This re-ordering is the
documented cause of the 5e-3 f64 band (an order of magnitude above the 1e-5
contract). The fix is mechanical and research-mandated.

GAP 1 (CR-01) then adds the missing `optimal`-schedule oracle test. The committed
fixtures `mbsgd_classifier_optimal_{f32,f64}_seed42.npz` are generated by
gen_oracle.py (lines 1832-1864) but NO Rust test loads them, so the Bottou `t0`
math and the default `SGDClassifier()` code path are entirely unvalidated against
sklearn.

Purpose: SGDSVM-01 (and ROADMAP Success Criterion #2 + the phase goal) require
MBSGDClassifier to match sklearn "schedules INCLUDING `optimal`". Without this the
headline default-schedule path is unproven.

Output: Re-ordered L2 shrink in prims/sgd.rs (with tightened/justified band), and
new `oracle_optimal` + `oracle_optimal_f32` tests in the classifier suite.
</objective>

    Reorder the per-batch body so the device weight gradient step
    (sgd_weight_update launch) runs FIRST on the current `w`, and the lazy-L2
    `wscale` shrink `w_j *= max(0, 1 - (1 - l1_ratio)·eta·alpha)` is applied to the

    release_into) and the `params.alpha > 0.0 && l2_factor != 1.0` guard; only the
    POSITION of the shrink moves, not its formula.

    Fix WR-02 in the same edit: snapshot the TRUE start-of-batch weights (the `w`
    BEFORE any update or shrink this batch) into a separate `w_start: Vec<f64>`
    vector, and compute the convergence delta `change = (w_new[j] - w_start[j]).abs()`
    against that snapshot — NOT against a `w_host` that has already been mutated by
    the penalty shrink. This makes `max_change` reflect the full per-batch update
    (gradient + penalty) so the `tol`-based early stop (the `params.tol > 0.0`
    block) does not fire prematurely. The committed fixtures pin `tol=0` so this is
    correctness-hardening for the user-facing `tol=1e-3` default path.

    Keep the convex-objective gate intact: `sgd_convex_objective` uses alpha≈0, so
    the L2 shrink is inert there (l2_factor==1.0) and the reorder is a no-op for it —

    the edit.
