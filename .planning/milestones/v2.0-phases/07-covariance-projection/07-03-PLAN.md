<objective>
Implement PRIM-07: the incremental-SVD merge primitive `prims/incremental_svd.rs`,
host glue that updates a running thin decomposition with a new batch by re-running the
v1 Jacobi `svd` on a small stacked matrix — EXACTLY scikit-learn's own IncrementalPCA
algorithm (dense re-SVD per batch, NOT a streaming rank-update). The `[v2-P1]` research
flag is SETTLED: full-Jacobi-per-batch reusing v1 svd, ZERO new device kernel
(RESEARCH §Summary + Pattern 1).

Purpose: the merge that IncrementalPCA.partial_fit (Plan 05) consumes. The stacked
operand is tiny ((k+b+1) × n_features), so the v1 Jacobi cost is negligible and its f32
stability is the SAME the v1 PCA path already passes its 1e-5 gate with.
Output: prims/incremental_svd.rs (merge + first-batch branch + svd_flip + ddof=1 +
cap validation); incremental_svd_test.rs green (2+-batch vs host reference + memory gate).
</objective>

  <action>
    Implement the host merge per RESEARCH Pattern 1 (exact sklearn 1.7.1 math), composing
    `prims::svd::svd` and `mlrs_core::sign_flip::align_rows`. Overwrite the Plan-01 stub
    body in incremental_svd.rs (do NOT re-edit prims/mod.rs).

    State carried: `(components_ [k×p, Vᵀ rows], singular_values_ [k], mean_ [p], var_ [p],
    n_samples_seen_)`. `merge::<F>(pool, state, x_batch, (b, p), n_components)`:

    1. `col_batch_mean = column_reduce(x_batch, Mean)` — this batch's OWN mean (BEFORE the
       running-stats update). Read it to host.
    2. Update running stats with Chan-Golub-LeVeque (`incremental_mean_var` host helper, in
       f64): produce `(col_mean, col_var, n_total)` from `(prev_mean, prev_var, n_seen,
       x_batch)`. (RESEARCH Pitfall 2: three distinct means — prior running `mean_`, batch
       `col_batch_mean`, updated running `col_mean`.)
    3. Center the batch by the UPDATED running `col_mean`: `X_c[r,c] = X_batch[r,c] − col_mean[c]`.
    4. BRANCH (Pitfall 3):
       - FIRST batch (k == 0 / no prior components): stacked = X_c alone (b × p).
       - SUBSEQUENT: build the (k + b + 1) × p host stack:
         rows 0..k     = `singular_values_[i] * components_[i, :]`  (host elementwise; = Σ·Vᵀ)
         rows k..k+b   = X_c
         row  k+b      = `sqrt((n_seen * b) / n_total) * (prev_mean − col_batch_mean)`  (mean_correction)
    5. VALIDATE the stacked shape against the SVD caps BEFORE the svd call (ASVS V5 / RESEARCH
       A2): require `(k + b + 1) ≤ MAX_ROWS (256)` and `p ≤ MAX_COLS (64)`; else return
       `PrimError::ShapeMismatch`. (svd.rs already validates at L430-441; validate here too so
       the error is attributable to the merge.) Import MAX_ROWS/MAX_COLS from mlrs-kernels.
    6. Upload the stack once (`DeviceArray::from_host`), call `svd::<F>(pool, &stacked,
       (k+b+1, p))` → `(U, S, Vᵀ)` with S descending.
    7. `align_rows` on the Vᵀ rows (== svd_flip u_based_decision=False) — apply after EVERY
       batch (Pitfall 5: sign stability across batches AND vs oracle).
    8. `explained_variance = S² / (n_total − 1)` (ddof=1 — Pitfall 1, DISTINCT from the
       covariance estimators' ddof=0); `explained_variance_ratio = S² / (sum(col_var) *
       n_total)` (Pitfall 6 — NOT the truncated-S² sum).
    9. Keep top `n_components`: store `components_ = Vt_flipped[:n_components]` device-resident,
       `singular_values_ = S[:n_components]`, `mean_ = col_mean`, `var_ = col_var`,
       `n_samples_seen_ = n_total`.
    10. Release the SVD scratch you don't keep (`u`, `s`, `vt`, the uploaded stack) via
        `release_into(pool)` — the memory gate depends on these releases.

    Accumulate ALL combine math in f64 (host_to_f64/f64_to_host — RESEARCH Pitfall 4
    stability convention). The whole merge is host glue + one device svd; no new kernel.
  </action>
