<objective>
Close GAP 2 (CR-02 BLOCKER): `Umap::fit` validates data-dependent geometry but never checks `n_components < n`. With the default `Init::Spectral` and small `n`, `run_umap_layout → spectral_init → spectral::recover` computes `let col = n - 1 - r` with `r` up to `n_components` (drop_first → `m = n_components + 1`); when `n_components + 1 > n` (e.g. `n=2, n_components=2`) this underflows `usize` → panic (debug) / wild OOB (release).

The fix mirrors the sibling `SpectralEmbedding::fit` guard (spectral_embedding.rs:155) which already rejects this case with the existing typed error `AlgoError::InvalidNComponents` (error.rs:41). The UMAP path simply omits the call. This is a one-guard fix plus a typed-error test — no kernel changes, no parallelism risk.

Purpose: Make `Umap::fit` robust on valid-typed input (criterion-1 robustness, UMAP-01) — a malformed-geometry guard BEFORE any device launch (ASVS V5 input validation).
Output: A guard in `Umap::fit` and a passing typed-error test.
</objective>
